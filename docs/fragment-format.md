# Fragment format

A fragment is one Markdown file under the source directory (default
`.rulecast/`) with YAML front matter. The front matter says *where* and
*for which tools* the rules apply; the body is the rules themselves,
passed through verbatim. Subdirectories inside the source dir are for
organization only â€” they do not affect scoping or naming. A `README.md`
inside the source dir is ignored.

```markdown
---
title: Web TypeScript conventions
description: Strict TypeScript rules for the web package
scope: packages/web/**
targets: [claude, cursor, copilot]
order: 20
---

- `strict` is on in tsconfig; never weaken compiler options in a PR.
- No `any` â€” use `unknown` plus narrowing.
```

## Front matter keys

| Key | Required | Default | Meaning |
|---|---|---|---|
| `title` | yes | â€” | Section heading in composed files; rule name in per-file targets. |
| `description` | no | falls back to `title` | One-liner surfaced in Cursor's `description` and Copilot's front matter. |
| `scope` | no | repo-wide | Glob limiting where the fragment applies. `"**"` means repo-wide. |
| `targets` | no | every configured target | Subset of `claude`, `agents`, `cursor`, `copilot`. |
| `order` | no | `100` | Sort key inside each compiled file; lower comes first, ties break by slug. |

Anything else is flagged by lint (`L105`) and ignored. The **slug** â€” the
filename minus `.md`, lowercased, punctuation collapsed to `-` â€” names the
per-fragment output files, so two fragments must never share one (`L106`).

## Scope globs

Supported syntax: `*` (within one path segment), `?` (one character),
`**` (any number of segments; must stand alone), and single-level
`{a,b}` alternation. Patterns are repo-relative; character classes,
absolute paths and `..` are rejected with an explanation (`L104`).

## How a scope maps to each tool

Claude Code and agents.md-aware tools scope rules by *file placement*
(the nearest `CLAUDE.md` / `AGENTS.md` up the tree); Cursor and Copilot
scope by *globs in front matter*. rulecast compiles to whichever
mechanism the tool actually has:

| Scope shape | claude / agents | cursor | copilot |
|---|---|---|---|
| none (repo-wide) | root `CLAUDE.md` / `AGENTS.md` section | `.mdc` with `alwaysApply: true` | section in `.github/copilot-instructions.md` |
| `dir/**` | **nested** `dir/CLAUDE.md` / `dir/AGENTS.md` | `.mdc` with `globs: dir/**` | `.instructions.md` with `applyTo: "dir/**"` |
| anything else (`**/*.sql`, `packages/*/src/**`, `{a,b}` â€¦) | root file section with an explicit "Applies to files matching â€¦" note | `.mdc` with the glob | `.instructions.md` with the glob |

The third row is the honest-compilation rule: a suffix or wildcard scope
cannot be expressed by directory placement, so rather than dropping the
information (or inventing a directory), the root file carries the glob as
a visible note, while Cursor and Copilot get it natively.

## The generated-file marker

Every output contains the HTML comment
`<!-- Generated by rulecast from <source>/ â€” do not edit by hand. â€¦ -->`.
That marker is how `build` knows a file is safe to overwrite (it refuses
otherwise, unless `--force`), and how `check` recognizes **orphans** â€”
marked files that no current fragment produces. Consequently the marker
phrase may not appear in a fragment body (`L110`).

## Lint rules

| Code | Severity | Fires when |
|---|---|---|
| L100 | error | front matter does not parse |
| L101 | error | `title` missing or empty |
| L102 | error | body is empty |
| L103 | error | unknown target name, or `targets` is not a list |
| L104 | error | scope glob is invalid (reason included) |
| L105 | warning | unknown front matter key (ignored) |
| L106 | error | two fragments share a slug |
| L107 | error | `order` is not a finite number |
| L108 | warning | scope's static directory prefix does not exist |
| L109 | error | `targets: []` â€” the fragment compiles to nothing |
| L110 | error | body contains the generated-file marker phrase |
| L111 | warning | two fragments share a title (case-insensitive) |
| L112 | warning | every requested target is disabled in rulecast.json |

Codes are stable API: never renumbered, never reused. Errors block
`build` and `check` (exit 2 â€” the plan cannot be trusted); warnings are
reported and, under `lint --strict` or `check --strict`, fail the run with exit 1.
